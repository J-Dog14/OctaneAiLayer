# 8ctane AI Layer

R&D pipeline that turns athlete assessment data + the entire history of coach-prescribed
programs into AI-generated weekly programs, ready to be inserted back into the Octane App DB.

The whole thing lives in the `ai_layer` schema of the backend Neon DB and is driven by
a single Click CLI (`src/main.py`).

## What this repo does

End-to-end flow, one athlete in → one weekly program out:

```
   App DB                            Backend DB (Neon, warehouse)
 ┌──────────┐                       ┌────────────────────────────────────────┐
 │ Programs │                       │ analytics.* (athlete dim + fact tables)│
 │ Exercises│  ─── sync ──────▶     │ app_db_snapshot.* (read-only mirror)   │
 │ Users    │                       │ ai_layer.* (the actual work)           │
 └──────────┘                       └────────────────────────────────────────┘
                                                       │
                                                       ▼
   ┌──────────────────────────────────────────────────────────────────────┐
   │ 1. assessment_norms        ← per-metric percentiles across athletes  │
   │ 2. athlete_profiles        ← per-athlete Z-scores across all domains │
   │ 3. program_summaries       ← structured summary of every App DB pgrm │
   │ 4. program_exercise_prescriptions ← every lift/plyo/prep/bp/me/hit   │
   │ 5. lift_templates          ← coach spreadsheet → typed catalog       │
   │ 6. recommended_programs    ← Gemini-generated weekly program (JSONB) │
   └──────────────────────────────────────────────────────────────────────┘
                                                       │
                                                       ▼
   ┌──────────────────────────────────────────────────────────────────────┐
   │ payload_builder            ← compiles recommended_programs row into  │
   │                              an App DB-compatible JSON payload that  │
   │                              mirrors WeeklyTemplate / ActivityTpl /  │
   │                              ExerciseToLift / PlyoToTemplate / etc.  │
   └──────────────────────────────────────────────────────────────────────┘
                                                       │
                                                       ▼
                          (future) API endpoint → Octane App DB inserts
```

The pipeline is fully re-runnable: every step is idempotent against its inputs, and every
LLM call is logged with version + cost so we can re-trace any recommendation later.

## What goes into a recommendation

A recommended program is built from three signals:

1. **The athlete's deficit profile** — Z-scores across pitching 3D (Qualisys), hitting 3D,
   mobility, Proteus, athletic screen (DJ / CMJ / PPU / SLV), and force plate. Anything
   below ≈ -0.5 Z is treated as a real deficit the program should address.
2. **The history of similar athletes** — for each component (prep, bp, plyo, lift, hitting),
   we pull the exercises coaches have historically prescribed to athletes with similar
   profiles. That history is the candidate pool the LLM picks from.
3. **The coach spreadsheet of lift templates** — ~510 templates encoded under a 4-7 digit
   ID scheme (e.g. `5227-11`) covering age group × focus × sport × archetype. The recommender
   picks one template per movement bucket (Legs / Upper / Total Body / Sprint / Jump).

Components produced:

| Component         | What it is                                                |
| ----------------- | --------------------------------------------------------- |
| **Prep**          | 6-10 exercises priming for the day. Med ball + waterbag work emphasized for Power/Speed athletes. Same every day. |
| **Bulletproofing**| Cuff + scap stability + iso work. Same every day.         |
| **Plyo**          | 3-day cycle (P0 / P1 / P2) gated by annual throwing phase. Reverse throws are always first, only one per session. |
| **Lift**          | 5 movement buckets from the spreadsheet (Legs / Upper / Total Body / Sprint / Jump). |
| **Hitting**       | Only for hitters.                                         |
| **Weekly layout** | Mon-Sun mapping of which lift bucket + which plyo day runs that day. |

Every component is generated by a focused Gemini 2.5 Flash call with a dedicated skill
(`skills/prep-programming/`, `skills/plyo-programming/`, `skills/lift-programming/`, etc.).
The whole thing is persisted as JSONB columns on `ai_layer.recommended_programs`.

## Payload compiler

Once a program is recommended, `src/payload_builder.py` compiles it into a JSON payload
shaped exactly like the App DB schema — one `WeeklyTemplate`, one `ActivityTemplate` per
component (prep / bp / plyo / hitting / 5 lift buckets), child rows for every
`ExerciseToLift` / `ThrowingExerciseToPlyoTemplate` / etc., a `WeeklyTemplateApplication`
linking the athlete, and a 7-day layout.

Exercise name → `Exercise.id` resolution happens here against `app_db_snapshot.Exercise`,
with a layered matcher:

- **exact** — case-insensitive equality (no review needed)
- **loose** — normalized (punctuation, `w/` vs `w `, curly quotes, etc.) equality (no review needed)
- **fuzzy:0.xx** — difflib similarity ≥ 0.88, flagged in the payload (`match_method`) so the
  downstream API endpoint can decide whether to require coach confirmation
- **unmatched** — listed in `payload.unmatched_exercises` with the top-3 nearest candidates
  for manual review

A future API endpoint owned by the Octane app consumes this JSON and does the actual inserts
into the App DB.

## Usage

### One-time / nightly maintenance

```bash
# Snapshot App DB into backend.app_db_snapshot.*
python -m src.main sync

# Snapshot a single table (for testing)
python -m src.main sync --tables User

# Refresh per-metric percentiles (assessment_norms)
python -m src.main norms

# Rebuild every athlete's Z-score profile
python -m src.main backfill

# Summarize every program in app_db_snapshot.Program into ai_layer
python -m src.main summarize-all

# Load the coach lift-template spreadsheet into ai_layer.lift_templates
python -m src.main load-templates
```

### Per-athlete

```bash
# Generate an athletic-screen analysis for one athlete/session
python -m src.main screen <athlete_uuid> 2026-06-15

# Generate a pitching biomech breakdown
python -m src.main biomech <athlete_uuid> 2026-06-15

# List the focus values you can pass to `recommend`
python -m src.main available-focuses

# Generate a full weekly program. Writes a row to ai_layer.recommended_programs,
# a markdown summary to outputs/, and the App DB payload JSON to outputs/.
python -m src.main recommend <athlete_uuid> --focus Power --phase Velocity

# Re-compile the App DB payload for an existing recommended program
python -m src.main compile-payload <recommended_program_id>

# Build a correlation report across the corpus
python -m src.main report
```

### Eval harness (recommender regression testing)

```bash
# Blind-eval the recommender against a coach's actual program for one athlete.
# Focus auto-detected from the coach's program goals; override with --focus.
python -m src.main eval-athlete "Connor Chicoli"

# Bulk eval — one athlete name per line in the file (# = comment)
python -m src.main eval-batch eval_athletes.txt

# Aggregate stats across every saved eval: per-component overlap averages,
# most-overprescribed exercises, most-missed exercises, lift template family
# distribution
python -m src.main eval-summary
python -m src.main eval-summary --focus Power
```

Each eval persists to `ai_layer.eval_runs` and writes a side-by-side markdown
report to `outputs/eval_*.md`. Cost is ~$0.02 per eval (one full recommend run);
$0.50 buys ~25 evals.

`recommend` supports `--focus`, `--plyo-day`, `--phase`, and `--athlete-role` flags so coaches
can override the defaults inferred from the athlete profile.

## Where outputs go

- **`ai_layer.recommended_programs`** — every recommendation, with JSONB columns for each
  component (`lift_program`, `prep_program`, `bulletproofing_program`,
  `movement_enhancement_program`, `plyo_program`, `hitting_program`)
- **`ai_layer.generated_reports`** — one row per assessment-driven report (athletic screen,
  biomech, etc.)
- **`ai_layer.athlete_profiles`** — current Z-score profile per athlete
- **`ai_layer.program_summaries` + `program_exercise_prescriptions`** — structured corpus of
  every App DB program (≈ 440 programs, 15-25k exercise rows)
- **`ai_layer.lift_templates` + `lift_template_exercises`** — coach-built spreadsheet catalog
- **`ai_layer.assessment_norms`** — per-metric percentiles across the population
- **`ai_layer.llm_call_log`** — every Gemini call with tokens + cost + latency + skill version
- **`ai_layer.prompt_versions`** — skill prompt history (so any past recommendation is
  reproducible)
- **`outputs/program_*.md`** — human-readable markdown summary
- **`outputs/payload_*.json`** — App DB-compatible payload for the future API endpoint

## Skills

Skills live in `skills/`. Each is a `SKILL.md` (the prompt) plus optional `references/`
markdown for deeper context. The recommender loads each skill via `prompt_loader.py`, which
hashes and version-pins the prompt in `ai_layer.prompt_versions` — so a year from now we can
look at any recommended program and replay exactly the prompt that produced it.

Current skill set:

- `athletic-screen-analysis` — DJ / CMJ / PPU / SLV breakdown
- `biomech-pitching-breakdown` — Qualisys / B_Young 8-cylinder report
- `prep-programming` — daily prep block (with med ball / waterbag emphasis for Power/Speed)
- `plyo-programming` — P0 / P1 / P2 / P3 selection + annual-phase gates + reverse-throw ordering
- `lift-programming` — coach spreadsheet template selection + progression
- `bulletproofing-programming` — cuff / scap / iso block
- `hitting-programming` — hitting program for non-pitchers

## Safety

- **Writes are warehouse-only.** All writes happen on the backend DB. The App DB connection
  is read-only (and in fact we read from the *snapshot* of it, never the live App DB).
- **Cost cap.** Every Gemini call checks a daily cost cap (`DAILY_COST_CAP_USD` in `.env`)
  before running.
- **Atomic snapshot.** `sync_app_db` uses a staging-table swap so partial-sync states are
  never visible to queries.
- **Reproducibility.** Every recommended program logs which skill version, model, and
  candidate pool it used. Re-running the same inputs against the same versions produces the
  same output.
- **Insertion safety.** The payload compiler never writes to the App DB. It produces JSON
  only. A future API endpoint owns the actual `INSERT` statements and is responsible for
  org-scoping and exercise-id validation. Fuzzy exercise matches are flagged so that endpoint
  can require coach review before insertion.
